# Experience — run viewer + terminal

Zero-dependency single-page app over intelligence-pipeline run outputs
(`intelligence/out/<run>/`) **and** the research corpus
(`research/*/forum`, `research/*/evidence`). Seven views behind a segmented
menu — overview, screen, triage, forum, axes, memo, network — plus a ⌘K terminal
palette, entity pages, and a settings drawer with a **pluggable skin
system**.

## Run

```sh
# from the repo root — any static server works, no build step
python3 -m http.server 8617
# landing page (architecture rundown + doors into everything):
#   http://localhost:8617/
# run viewer directly:
#   http://localhost:8617/experience/?run=ferrite-inverted&view=overview
```

The landing page is `/index.html` at the repo root ([landing.css](landing.css));
it reuses skeleton + app chrome + skins unchanged, and the topbar brand in the
viewer links back to it.

`?run=<dir>` selects any directory under `intelligence/out/`; missing files
hide their sections. `?view=` deep-links a view. Keys 1–7 switch views.
**Replay run** (settings drawer) re-reveals the overview on the run's own
`latency.json` clock, compressed to ~9 s.

## Terminal: entities × functions

The Bloomberg-terminal interaction *pattern* (entity keys × function
mnemonics, keyboard-first), clean-room reimplemented over vc-brain's own
corpus — persons, forums, evidence signals, runs, docs. No market data.

- **⌘K / Ctrl-K / `/`** opens the palette. Type an entity
  (`eghbal` (AI-simulated persona), `gsig-026`, `gbrain`), optionally
  followed by a function:
  `eghbal pre`, `gbrain adj`, `net`, `help`. Fuzzy-matched, ranked,
  arrow-keys + enter.
- **Functions**: `DES` profile · `PRE`/`POST` interview docs (person) ·
  `DEB`/`ADJ` debate + adjudication (forum) · `NET` network graph ·
  `OPEN` run · `HELP`.
- **Entity pages** (`?view=entity&e=person:nadia-eghbal&fn=DES` — an
  AI-simulated persona) are fully
  deep-linkable; every `gsig-*` chip and person name is a link, so any
  claim is one click from its evidence — the legitimacy contract.
- **Network** (view 6, or `NET`) is a hand-rolled SVG force layout:
  forums · attendants · suggested attendants (dashed) · cited signals.
  Drag to move, wheel to zoom, click to open. The signals toggle strips
  the evidence layer.
- **Corpus manifest**: [corpus.json](corpus.json), regenerated by
  [build-corpus.py](build-corpus.py) (stdlib-only scan of
  `intelligence/out/*` + `research/*/forum`); the committed file is the
  no-build fallback. Everything else is fetched client-side at boot from
  the forum's own `seed.json` / `adjudication.json` / `evidence.json`.
- **Modules**: [terminal.js](terminal.js) index + palette + grammar +
  router · [entity.js](entity.js) person/forum/signal/doc pages ·
  [graph.js](graph.js) force layout. All skeleton-first: structure in
  app.css, no new material, skins apply unchanged.

## Architecture: skeleton + mounted skins

Follows the skeleton/skin precedent from the author's private gallery
system (skeleton.css, skin drivers):

- **[skeleton.css](skeleton.css)** — the reveal-0 reference: page wash, 1px
  outlines, glyph ink, blue-only reactivity. No material. This is the
  shipped look when no skin class is present, and the permanent fallback.
- **[app.css](app.css)** — structure only: the `body[data-view-state]` view
  machine (View Transitions slide, direction from a rank map, topbar frozen
  via `view-transition-name`), kleos-style segmented menu, settings drawer,
  content layout, overflow contract (`overflow-x: clip` + wrap-anywhere +
  `min-width: 0`).
> **Operator lock (2026-07-21, bead vc-brain-toe.5):** the viewer and landing
> are locked to **dark aluminium** — `html.skin-alu.theme-dark` is stamped
> unconditionally, the skin/theme switching UI is removed from the settings
> drawer and landing, and `glass-driver.js` never mounts. The skins/*.css and
> the driver stay on disk (this ledger references them); only the switching is
> gone. The description below documents the retired switchable system.

- **skins/** — each skin is scoped under an `html.skin-*` class, formerly
  stamped **before first paint** from `localStorage['vcbrain-skin:v1']`
  (gallery skeleton-mode.js pattern) and switchable live from the settings
  drawer (both now removed by the operator lock above):
  - **[skins/alu.css](skins/alu.css)** — flat-CSS aluminium via the
    machining law (`hsl(from …)` stops off per-theme base colours).
  - **[skins/glass.css](skins/glass.css)** + **[glass-driver.js](skins/glass-driver.js)**
    — *Melt Dichro Blue*, ported from farm variant
    `fc145-melt-dichro-blue.frag`. The driver compiles the variant's
    `shade()` verbatim over a mini harness (SHARED helpers + a DOM-rect
    scene + the FOOTER Surf assembly) on one fixed full-viewport WebGL2
    canvas; chrome elements tagged `data-skin="bar|chip|btn|face"` are
    sampled per frame into rounded-rect SDF regions (hairline chamfer 1.4px
    → kind 2 rims), active state drives the 68% accent mix. On WebGL2
    success the driver stamps `html.glass-live` and the chrome's CSS
    dressing steps aside; otherwise the CSS glass in glass.css (hatch
    6.5px @120°, contours 182px, melt 9px/s, dichroic conic rim keyed to
    `--scroll`, band flash σ=26px) stands alone.

Adding a skin = one scoped CSS file (+ optional driver with `mount()` /
`unmount()`), one chip in the settings drawer. The skeleton is never
edited.

The skeleton/skin architecture and both skins are adapted from the author's private gallery system.

## Fonts & support

Source Serif 4 + Space Mono (Google Fonts, needs network). `hsl(from …)`
and View Transitions need Chrome 119+/Safari 18+; both degrade gracefully
(flat colours / hard swaps). WebGL2 absent → CSS glass fallback.
